| M1 | Brand profile onboarding | Build an onboarding flow that allows new customers to quickly set up their brand profile in the system. Instead of manual configuration, the system automatically extracts key brand information from inputs such as a website URL, product data, images, or example ads. The user reviews and edits these suggestions in a simple interface before saving. The final output is a reusable "Brand Profile" used by the system to generate ads, select personas, and guide tone and layout decisions. Goal: reduce manual setup, improve ad quality, and make the system scalable across multiple customers. Step 1 (URL extraction + review form) is done on feat/m1-brand-onboarding branch. | in progress | — |
| F17 | Grid review mode | A gallery-style view for reviewing all generated ads at once — essential for high-volume ecommerce customers. Instead of the current single-image detail view, shows all ads as thumbnails in a grid. User scans quickly: hover to preview full size, click to select/deselect, X to reject without opening. Bulk approve selected ads in one click. Filtering by style family, product, and language. "Select all" and "Select none" shortcuts. Clicking a thumbnail opens the existing detail view for fine-tuning, then returns to grid. This replaces one-by-one approval for customers managing 100+ active ads. | open | — |
| F18 | Bulk export | Export all approved ads in a single zip with an organized folder structure: product → language → format. Every approved ad exported across all its language variants and formats at once — not one by one. For large catalogs (50+ products × multiple styles × multiple formats) the export runs as a background job with a progress indicator rather than a synchronous download. Optional: include a CSV manifest alongside the zip (product name, format, language, template ID) for media buyers uploading to Meta Ads Manager. Future extension: direct export to Meta Ads Manager via their API, skipping the download entirely. | open | — |
| F19 | Direct export to Meta Ads Library | Push approved ads directly to the customer's Meta (Facebook/Instagram) Ads Manager media library without downloading. User connects their Meta Business account via OAuth once. On export, approved PNGs are uploaded directly to their Meta asset library using the Marketing API — ready to use in campaigns immediately. No zip, no manual upload, no folder hunting. Could support tagging assets with product name, format, and language on upload so the media buyer can filter inside Meta. Works for both single ads and bulk-approved batches. Requires Meta App review approval for the `ads_management` permission. | open | — |
| F16 | Save AI-generated layouts as reusable templates | After generating an AI Style ad the user likes, they can save the layout for future use. The AI-generated HTML/SVG is stored in the DB stripped of the specific product photo, keeping only the composition structure (panels, colors, typography, decorative elements). Saved layouts appear as new pills in the layout picker alongside the built-in ones, clearly marked as "My Layouts" or similar. When selected for a new image, the saved layout re-renders with the new product photo inserted into the correct position. Users can name and delete saved layouts. A small layout library panel shows all saved layouts as thumbnails. This turns one-off AI generations into a growing personal template library that gets more valuable over time. | open | — |
| F15 | AI Style post-generation editing | After the AI Style (Surprise Me) ad is generated, allow the user to make light adjustments before saving/baking. Currently the result is final with no controls. Feasible edits to explore: (1) **Text editing** — click any headline or subtext in the rendered ad to change the copy in-place, re-render with updated text; (2) **Product image pan/crop** — drag the product photo within its container to reframe it, or adjust zoom level; (3) **Accent color tweak** — if the AI used a single accent color (border, panel, overlay), expose a color picker to override it without full re-generation; (4) **Headline font swap** — swap the headline font from a small set of options. Full layout changes (panel structure, composition) are NOT in scope — those require a new AI generation. (5) **Remove decorative elements** — the AI sometimes renders symbols, icons, ornamental shapes, or decorative lines that don't look right for a given brand. The user should be able to click any such element to select it and press Delete (or a trash icon) to remove it before the final bake. This requires rendering the ad in an interactive iframe, making individual SVG/HTML elements clickable, and patching the HTML to remove the selected node before re-screenshotting. Implementation note: the AI output is an HTML/SVG string rendered by Playwright. Edits would either (a) patch the HTML string before re-screenshotting, or (b) render interactively in an iframe and let the user manipulate/delete elements live before a final screenshot. Option (b) is more powerful and covers all edit types including element deletion. | open | — |
| F14 | Add more images to existing session | While a folder is loaded and images are analyzed, the user can add more images without clearing the current queue. Three entry points: (1) a persistent "Add more" / "+" button in the left panel below the image list, (2) drag-and-drop of image files or a folder directly onto the left panel or the main canvas at any time, (3) clicking the original folder picker again appends rather than replaces. New images join the bottom of the queue and go through the normal Analyze flow. Already-analyzed images are unaffected. Drop zone highlights with a visible border/overlay when files are dragged over the window. | open | — |

Optional: a subtle magnetic snap so the element locks to center with a small pull, making it easy to place exactly. Works for all draggable elements on the stage. | open | — |

| OPS2 | Persistent image storage | Currently generated PNGs are written to the local filesystem which is ephemeral on Vercel — files can disappear between deployments or serverless function invocations, meaning approved ads could vanish for a live customer. Fix: store all generated PNGs in a persistent blob storage service. Best options: (1) **Vercel Blob** — simplest, already integrated with Vercel, pay per GB, minimal code change; (2) **Cloudflare R2** — cheaper at scale, S3-compatible API; (3) **AWS S3** — most flexible, most setup. Change would affect the generate/reposition routes (write to blob instead of disk) and the image serving routes (read from blob URL instead of local path). Uploaded product photos should also move to blob storage for the same reason. Critical to solve before any customer goes live on Vercel. | open | — |
| OPS1 | New customer deployment checklist | Steps to spin up a new customer deployment. (1) Create a fresh Turso DB: `turso db create [customer-name]-ads` then `turso db show [customer-name]-ads --url` and `turso db tokens create [customer-name]-ads` to get the URL and auth token. (2) Go to Vercel → duplicate the existing project → set env vars: `TURSO_URL`, `TURSO_AUTH_TOKEN`, `ANTHROPIC_API_KEY`, `NEXT_PUBLIC_APP_URL` (their new deployment URL). (3) Deploy. (4) Visit `/api/onboarding/reset` on the new URL to confirm DB is clean. (5) Set a customer password once F24 (login) is built. (6) Hand over the URL. Until multi-tenancy (F20) is built each customer gets their own Vercel project and Turso DB. Turso free tier supports 500 databases so cost is zero for the first many customers. Target: 15-minute repeatable process per new customer. | open | — |
| F20 | Multi-tenancy / customer management | One codebase serving multiple customers with fully isolated data. Every DB record tagged with a `customer_id`. A simple admin panel to create customers, toggle feature flags per customer, and manage their brand profiles. Feature flags stored per customer in a JSON config — flip a flag and their app changes without redeployment. Each customer gets their own login credentials. Hosting options: one shared deployment (cheapest) or one deployment per customer (full isolation). Foundation for scaling the product across many customers without maintaining separate codebases. | open | — |
| F21 | Product naming and tagging | Let users name and tag their product images instead of relying on filenames. Each image in the queue gets an editable product name, optional SKU, and collection/category tag. Names and tags are shown in the queue list, used in export folder structure, and passed to Meta on upload. Filtering and sorting the queue by collection or tag. Essential for customers with large catalogs where filenames like "IMG_4521.jpg" are meaningless. | open | — |
| F22 | Ad performance notes | Let users mark approved ads with performance feedback — e.g. "performed well", "low CTR", "paused". Notes stored per ad in the DB alongside the template and copy used. Over time this builds a dataset of what works for their brand: which styles, headlines, and personas drive results. Future use: feed this signal back into generation to bias toward proven combinations. | open | — |
| F23 | Session persistence | Currently the image queue and all work in progress may be lost on page refresh. Persist the full session state to the DB so users can close the browser and return exactly where they left off — same images, same generated ads, same approved items. Critical for large catalogs where a session represents hours of work. Approved and baked PNGs are already stored; this extends persistence to the full queue state including pending and analyzed items. | open | — |
| F24 | User accounts and login | Basic authentication so only authorized users can access the app. Minimum viable: a simple password-protected login page with a session cookie. Full version: named user accounts with email + password, so multiple team members at a customer can log in separately. Needed before sharing any deployment with a customer. Could use NextAuth.js for quick implementation. Ties into multi-tenancy (F20) — each login maps to a customer account. | open | — |
| F25 | Undo | Undo the last action across all editing operations — deleting a text box, moving a headline, approving an ad, changing a color, hiding an element. Ctrl/Cmd+Z to undo, Ctrl/Cmd+Shift+Z to redo. Implemented as an action history stack in React state. Scope: canvas editing actions only (not analyze or generate). Prevents the frustration of accidentally deleting work with no way to recover. | open | — |
| F26 | Keyboard shortcuts | Full keyboard shortcut set for power users managing large ad batches. Core shortcuts: approve selected ad, reject/unapprove, next/previous image in queue, next/previous style, new headline, open translate, open export. Shortcut cheat sheet accessible via "?" key or help panel. Especially valuable combined with grid review mode (F17) — lets a media buyer process 100 ads in minutes without touching the mouse. | open | — |
